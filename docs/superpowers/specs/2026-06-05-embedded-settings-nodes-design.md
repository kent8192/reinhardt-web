# Embedded Settings Nodes and Typed Schema References

## Summary

Issue #5176 adds first-class settings schema nodes for reusable nested
configuration structs. The design keeps root settings composition compatible
while allowing nested structs to participate in generated schema metadata,
recursive required-field validation, and secret-field discovery without forcing
their nested TOML path to come from a root section name.

The main public contract is a new typed schema layer. Root fragments and
embedded nodes both implement `SettingsNode`; composed settings expose a
fluent, type-safe schema accessor so callers can refer to fields without
hand-written string paths.

## Goals

- Allow `#[settings(fragment = true)]` on reusable settings structs, including
  structs reused below other fragments.
- Keep existing `SettingsFragment`, `HasSettings<F>`, and `Has*Settings`
  behavior for root-composable fragments.
- Treat `section = "..."` as a root-composition default key hint. When a
  fragment is embedded, the parent field name controls the TOML path component.
- Generate typed fluent field references for root and nested settings fields.
- Discover secret fields recursively for `SecretString` and `SecretValue<_>`.
- Validate missing required fields recursively before deserialization.
- Preserve existing root-fragment compatibility and root-only error behavior.

## Non-Goals

- Do not replace `SettingsFragment` with `SettingsNode`.
- Do not support arbitrary generic wrapper introspection.
- Do not support `serde(flatten)` inside settings nodes.
- Do not add custom user-defined secret type classification in this change.
- Do not remove existing `section()` or `field_policies()` APIs.

## User-Facing Shape

Users can define a root fragment and nested embedded node with the same
attribute family:

```rust
#[settings(fragment = true, section = "database")]
pub struct DatabaseSettings {
    pub default: DatabaseConfig,
    pub replica: Option<DatabaseConfig>,
}

#[settings(fragment = true)]
pub struct DatabaseConfig {
    pub engine: DatabaseEngine,
    pub host: String,
    pub port: u16,
    pub name: String,
    pub user: String,
    pub password: SecretString,
}

#[settings(database: DatabaseSettings)]
pub struct ProjectSettings;
```

The resulting TOML uses the composition key for the root path and parent field
names for embedded path components:

```toml
[database.default]
engine = "postgresql"
host = "localhost"
port = 5432
name = "app"
user = "app"
password = { env = "DATABASE_PASSWORD" }

[database.replica]
engine = "postgresql"
host = "replica.internal"
port = 5432
name = "app"
user = "readonly"
password = { file = "/run/secrets/db-replica-password" }
```

The typed schema API exposes fluent references:

```rust
let schema = ProjectSettings::schema();

let default_password = schema.database.default.password;
let replica_password = schema.database.replica.some().password;

assert_eq!(default_password.path().to_string(), "database.default.password");
assert_eq!(replica_password.path().to_string(), "database.replica.password");
```

Secret-only APIs accept only `SecretFieldRef`:

```rust
fn resolve_secret(field: SecretFieldRef<ProjectSettings, SecretString>) {
    // Secret resolution uses the rendered settings path for diagnostics.
}

resolve_secret(ProjectSettings::schema().database.default.password);

// This does not compile because `host` is not a secret field.
// resolve_secret(ProjectSettings::schema().database.default.host);
```

## Public API Additions

Add a schema module under `reinhardt_conf::settings` with these core concepts:

```rust
pub trait SettingsNode {
    fn node_schema() -> SettingsNodeSchema;
}

pub trait HasSettingsSchema {
    type Schema;

    fn schema() -> Self::Schema;
}

pub struct FieldRef<Root, Value> {
    // Path metadata plus root/value type markers.
}

pub struct SecretFieldRef<Root, Value> {
    // Path metadata plus root/value type markers.
}

pub struct SettingsPathBuf {
    // Ordered path segments, renderable for diagnostics.
}

pub enum SettingsPathSegment {
    Key(&'static str),
    DynamicKey(String),
    AnyKey,
    AnyIndex,
}
```

`SettingsFragment` remains the root-composition compatibility trait. Every
struct generated by `#[settings(fragment = true)]` implements `SettingsNode`.
It also keeps the existing root-fragment surface: `SettingsFragment`,
`section()`, `HasSettings<Self>`, and `Has*Settings`. The difference is that
`section()` becomes a default root key hint. When the same type is embedded
below another node, the embedded path is owned by the parent field name instead
of by the embedded type's section hint.

## Section and Path Semantics

Root TOML keys come from the composition entry:

```rust
#[settings(database: DatabaseSettings)]
pub struct ProjectSettings;
```

The root path is `database`.

If composition uses type-only syntax, `section = "..."` acts as the default key
hint:

```rust
#[settings(fragment = true, section = "database")]
pub struct DatabaseSettings;

#[settings(DatabaseSettings)]
pub struct ProjectSettings;
```

The root path is `database`.

If `section = "..."` is omitted, type-only root composition infers the key from
the fragment type name:

```rust
#[settings(fragment = true)]
pub struct DatabaseSettings;

#[settings(DatabaseSettings)]
pub struct ProjectSettings;
```

The root path is `database`.

If an explicit key and `section` disagree, the explicit composition key wins:

```rust
#[settings(fragment = true, section = "database")]
pub struct DatabaseSettings;

#[settings(db: DatabaseSettings)]
pub struct ProjectSettings;
```

The root path is `db`.

Embedded TOML path components come from parent field names or their
`serde(rename = "...")` values, not from the embedded node type name or section
hint.

```rust
#[settings(fragment = true)]
pub struct DatabaseSettings {
    pub default: DatabaseConfig,
}

#[settings(fragment = true, section = "connection")]
pub struct DatabaseConfig {
    pub password: SecretString,
}
```

When `DatabaseSettings` is mounted as `database`, the nested password path is
`database.default.password`, not `database.connection.password` and not
`database.database_config.password`.

## Container Traversal

The schema generator unwraps semantically agnostic containers. These wrappers do
not change the semantic settings value; they only affect traversal:

```rust
Option<T>              // .some()
Vec<T>                 // .any()
HashMap<String, T>     // .entry(key), .any()
BTreeMap<String, T>    // .entry(key), .any()
IndexMap<String, T>    // .entry(key), .any()
Box<T>                 // transparent
```

Examples:

```rust
schema.database.replica.some().password;
schema.database.tokens.any();
schema.core.databases.entry("default").password;
schema.core.databases.any().password;
schema.database.boxed.password;
```

Unsupported wrappers such as `Result<T, E>`, `Cow<'a, T>`, `Arc<T>`, `Rc<T>`,
`Mutex<T>`, `RwLock<T>`, and `PhantomData<T>` are treated as leaf fields. Their
inner types are not inspected for required fields or secrets.

## Serde Attribute Handling

`serde(rename = "...")` is reflected in rendered settings paths while preserving
Rust field names in the fluent API:

```rust
#[settings(fragment = true)]
pub struct ApiConfig {
    #[serde(rename = "api-key")]
    pub api_key: SecretString,
}

let field = ProjectSettings::schema().service.api_key;
assert_eq!(field.path().to_string(), "service.api-key");
```

`serde(flatten)` is a compile error on any field inside a generated settings
node. Flattening erases ownership of path components and makes recursive
required validation and secret diagnostics ambiguous.

## Recursive Required Validation

`SettingsBuilder::build_composed::<T>()` validates required fields against the
merged raw settings value before deserialization:

```rust
let merged = builder.build()?;
ProjectSettings::validate_schema_requirements(merged.as_map())?;
let settings = merged.into_typed::<ProjectSettings>()?;
```

Missing nested required fields return a new non-breaking error variant:

```rust
BuildError::MissingRequiredPath {
    path: SettingsPathBuf::from_segments([
        SettingsPathSegment::Key("database"),
        SettingsPathSegment::Key("default"),
        SettingsPathSegment::Key("password"),
    ]),
}
```

The existing `BuildError::MissingRequiredField { section, field }` variant
remains for root-only compatibility and existing callers.

## Secret Discovery

The schema tree marks fields as secret when their unwrapped type is
`SecretString` or `SecretValue<_>`. The same container whitelist used for
traversal applies to secret detection, so `Option<SecretString>`,
`Vec<SecretString>`, maps of secret values, and boxed secret nodes are detected.

Discovery returns typed refs and renderable paths:

```rust
let secrets = ProjectSettings::schema().secret_fields();

assert!(secrets.iter().any(|field| {
    field.path().to_string() == "database.default.password"
}));
```

For dynamic maps, discovery exposes templates:

```rust
let template = ProjectSettings::schema().core.databases.any().password;
assert_eq!(template.path().to_string(), "core.databases.*.password");
```

Callers can also create concrete refs for runtime keys:

```rust
let concrete = ProjectSettings::schema()
    .core
    .databases
    .entry("default")
    .password;

assert_eq!(concrete.path().to_string(), "core.databases.default.password");
```

## Macro Generation Strategy

The settings fragment macro should share its field parsing logic between root
fragments and embedded nodes. It should generate:

- Derived serde/debug/clone behavior as before.
- Field policy metadata for the node itself.
- `SettingsNode` implementation for all generated nodes.
- `SettingsFragment`, `HasSettings<Self>`, and `Has*Settings` for generated
  fragments, with `section()` returning the explicit section hint or the
  inferred type key.
- Schema accessor structs for composed settings roots and nested nodes.

The composition macro should generate `HasSettingsSchema` for the composed
settings type and use the composed root keys to seed the schema path.

## Tests

Add focused tests for the new behavior:

- Compile-pass: `#[settings(fragment = true)]` without `section` compiles as an
  embedded node.
- Compile-pass: fluent typed refs compile for root, nested, optional, dynamic,
  and transparent boxed nodes.
- Compile-fail: a non-secret field cannot be passed to a `SecretFieldRef` API.
- Compile-fail: `serde(flatten)` inside a settings node is rejected.
- Integration: `build_composed()` returns `MissingRequiredPath` for nested
  required fields.
- Integration: secret discovery sees `SecretString` and `SecretValue<_>` behind
  `Option`, `Vec`, `HashMap`, `BTreeMap`, `IndexMap`, and `Box`.
- Regression: existing root fragments still implement `SettingsFragment`,
  `HasSettings<Self>`, `Has*Settings`, and `field_policies()`.
- Regression: explicit composition keys continue to override section hints.

## Acceptance Criteria

- Existing `#[settings(fragment = true, section = "...")]` fragments keep
  compiling and keep their current root-fragment behavior.
- Nested settings structs can use `#[settings(fragment = true)]` and expose
  schema metadata without forcing embedded TOML paths to use root section names.
- Composed settings expose typed fluent field references.
- Recursive required validation reports nested missing fields with
  `MissingRequiredPath`.
- Secret discovery exposes typed refs for supported secret fields and container
  shapes.
- Documentation and tests describe section hints, embedded field path semantics,
  container traversal, and unsupported wrappers.
