# Reinhardt 0.4 Performance Scorecard

This document defines the performance contract for the `develop/0.4.0`
breaking-change line. The goal is to remove compatibility costs from the
default HTTP, routing, and compile-time paths while keeping migration support
outside the hot path.

## Scope

The 0.4 performance track targets these surfaces:

- Runtime HTTP loopback request handling.
- Native endpoint and `server_fn` dispatch.
- Request allocation count and tail-latency probes.
- Minimal compile-time fixture builds.
- Feature graph size for server-only applications.

Synthetic database, contract, and admin matrix scenarios remain useful for
coverage, but they are not the primary proof that framework runtime paths are
faster.

## Baseline

Use the committed 2026-06-26 framework comparison report as the initial
planning baseline. Re-run the full scorecard before merging any performance
claim because local loopback measurements can be noisy.

Runtime HTTP promotion uses absolute latency ranges and framework parity, not a
fixed percentage reduction from 0.3. The loopback benchmark includes client,
socket, Hyper, scheduler, and OS costs; once a host reaches its loopback floor,
small framework hot-path wins are no longer visible as stable relative
percentages.

## Measurement Contract

Runtime comparisons must use loopback HTTP, not in-process router-only paths.
Each scorecard run should record at least three runs and report the median of
the point estimates.

```bash
cargo make benchmark-suite-check
cargo make benchmark-suite-measure
cargo make benchmark-runtime-http
cargo make benchmark-matrix-measure
cargo bench -p reinhardt-pages --bench server_fn_endpoint_benchmarks -- --sample-size 30 --measurement-time 2
cargo run --release -p reinhardt-benchmarks --bin request_alloc_probe
cargo run --release -p reinhardt-benchmarks --bin request_latency_percentile_probe
```

When a full run is too expensive, the PR may include a focused subset, but it
must state which scorecard dimensions were not refreshed.

measurements too noisy to trust. The remote host must be recorded with the
result because the absolute latency ranges above are host-class-specific.
Compare target ranges only against measurements taken on the same host class as

## Breaking Change Inventory

The 0.4 line may intentionally break the following APIs when doing so removes
default hot-path costs.

| Area | Breaking direction | Performance reason |
| --- | --- | --- |
| HTTP request fields | Replace direct public field access with accessor/extractor APIs. | Enables lazy query parsing, lazy extension storage, and internal layout changes. |
| Query parameters | Replace eager `HashMap<String, String>` construction with lazy or typed extraction. | Avoids allocation on routes that do not use query parameters. |
| Path parameters | Replace always-owned `String` storage with borrowed or typed extraction. | Avoids per-request parameter string allocation and repeated lookup parsing. |
| Extensions | Remove shared clone semantics from the default request-local store. | Avoids `Arc<Mutex<HashMap<...>>>` allocation for requests that do not use extensions. |
| Handler trait | Add a typed service path and move boxed `async_trait` dispatch to compatibility adapters. | Avoids boxed futures and dynamic dispatch for generated endpoints. |
| Middleware trait | Add a typed middleware path for generated/server-owned stacks. | Lets empty and single-middleware stacks avoid vector and trait-object overhead. |
| Feature defaults | Make minimal server features the default for server runtime crates. | Keeps parser, page, reactive, macro-heavy, and optional integration dependencies out of minimal builds. |

Compatibility adapters may exist behind a feature such as `legacy-0-3`, but the
default 0.4 runtime path must not pay for them.

## Implemented 0.4 API Breaks

The first HTTP request-model and parameter-extractor changes intentionally break
these 0.3-era surfaces:

- `Request::query_params` is a lazy `QueryParams` wrapper. Call `as_map()` or
  `to_hash_map()` when owned map behavior is required.
- `QueryParams::get()` returns `Option<&str>` and does not materialize the
  parsed map.
- Empty request extensions are not eagerly allocated or shared by
  `Request::clone_for_di()`. Initialized extension stores remain shared.
- `PathParams::get()` returns `Option<&str>`.
- `PathParams::iter()` yields `(&String, &str)`.
- `PathParams::values()` yields `&str`.
- `PathParams::as_slice()` is removed; use `to_vec()` for owned pairs.
- `PathParams::insert()` and `Request::set_path_param()` accept borrowed string
  values.
- Server route compilation now freezes method routers in an immutable
  `OnceLock` table and keeps a method-indexed exact-route table for static
  paths. This is not a public API break, but it removes per-request route-table
  read locks from dispatch and lets parameter-free routes skip matchit lookup
  and empty path-parameter materialization.
- `SyncHandler`, `ServerRouter::{endpoint_sync, handler_sync}`, and
  `ServerRouter::try_dispatch_sync` provide a synchronous route fast path.
  Routes without middleware can call this handler directly instead of
  allocating the boxed future required by the async `Handler` compatibility
  adapter.
- `RequestlessSyncHandler`,
  `ServerRouter::{endpoint_requestless_sync, handler_requestless_sync}`, and
  `ServerRouter::try_dispatch_requestless_sync` provide a request-free route
  fast path. HTTP/1 adapters can execute eligible empty-body routes before
  constructing a full `Request`; middleware, path-parameter, and DI routes
  remain on the request-shaped path.
- Exact `SyncHandler` and `RequestlessSyncHandler` routes bypass generic
  `RouteMatch` construction on the HTTP adapter fast path when the router has
  no child router or middleware stack to inherit.
- Native `ServerFnHandler` returns serialized `bytes::Bytes` instead of
  `String`. Generated `#[server_fn]` handlers serialize JSON responses with
  `serde_json::to_vec()`, and server-function responses use static typed
  `Content-Type` headers plus direct JSON body response construction on the
  dispatch path.
- Generated JSON server-function handlers borrow the request body directly when
  no extractor or DI parameter needs body-consumption tracking after decoding.
  This no-extractor/no-DI JSON path no longer content-negotiates non-JSON
  request bodies; use `codec = "url"` for URL-encoded server-function payloads.
- Generated `ServerFnRegistration::handle()` implementations return their
  concrete handler future, avoiding the server-function wrapper box before
  endpoint response assembly. The legacy `ServerRouter::handle()` benchmark
  still pays the outer `Handler` trait-object box for the route endpoint.
- `ServerFnMetadata::IS_JSON_CODEC` records generated codec selection as an
  associated constant so native response assembly can select the JSON fast path
  without a per-request string comparison.
- Server response conversion skips redundant status/header mutation for the
  common `200 OK` response without headers, release builds skip non-API warning
  path scans on successful requests, and raw query lookup scans cached pairs
  from the end so last-value duplicate semantics can return early.

See [`../migration/0.4.0-performance-api.md`](../migration/0.4.0-performance-api.md)
for the migration guide.

## Refactoring Phases

### Phase 1: HTTP Request Model

Replace eager request normalization with a lazy request model:

- Skip body collection for empty `GET` and `HEAD` requests.
- Parse query parameters only when an extractor asks for them.
- Allocate extensions only after the first insertion.
- Keep JSON request bodies as bytes until deserialization.

Expected result: 10-20% runtime HTTP improvement on lightweight scenarios.

### Phase 2: Frozen Router Table

Move route compilation fully out of the per-request path:

- Compile router tables at startup or registration finalization.
- Remove per-request `compile_routes()` checks.
- Replace read-locked router maps with immutable compiled tables.
- Keep Django-style trailing-slash fallback allocation-free on the common path.

Expected result: 8-18% improvement on route-heavy scenarios.

### Phase 3: Typed Service Path

Introduce a typed service path for generated endpoints:

- Keep `Arc<dyn Handler>` only as a compatibility adapter.
- Use associated futures or concrete service types for generated endpoints.
- Preserve middleware composition without allocating a chain for empty stacks.

First implemented step: `SyncHandler` covers endpoints that complete without
awaiting I/O and keeps `Arc<dyn Handler>` as the middleware/compatibility
adapter.
Second implemented step: `RequestlessSyncHandler` covers generated endpoints
with no request-shaped inputs and lets the HTTP/1 adapter skip `Request`
construction for empty-body exact routes.
Third implemented step: generated native server-function handlers keep
serialized responses as bytes and avoid parsing static `Content-Type` headers
while building the endpoint response.
Fourth implemented step: generated JSON server-function handlers borrow request
bodies directly when later extractors do not need body-consumption state.
Fifth implemented step: generated server-function registrations override
`ServerFnRegistration::handle()` with a concrete future while keeping the boxed
`handler()` compatibility entry point.

Expected result: 10-25% runtime HTTP improvement where boxed futures dominate.

### Phase 4: Parameter Extractors

Move path and query extraction from map construction to typed extraction:

- Borrow path captures while the request is handled.
- Parse directly into requested extractor types.
- Keep owned conversion available for compatibility adapters.

Expected result: 10-20% improvement for `path_params` and `query_params`.

### Phase 5: Minimal Feature Graph

Split feature defaults and crate dependencies around server runtime needs:

- Keep parser-heavy and page/reactive features opt-in.
- Allow server-only router usage without client/router macro dependencies when
  macros are not used.
- Keep compile-time comparison fixtures aligned with the public minimal path.

Expected result: 25-40% clean minimal build improvement and 30-50% incremental
check improvement.

## Promotion Gates

The 0.4 performance branch can be promoted when:

- Runtime HTTP medians are inside the 0.4 promotion target ranges listed above
  for the primary runtime scenarios on a low-noise host. Any miss must be
  within 5% of the target or documented as host-floor noise.
- Runtime HTTP parity is checked across all measured runtime scenarios,
  including `middleware_chain`; Reinhardt should be no more than 5% slower than
  the fastest comparison framework in at least five measured scenarios on the
  same host. Larger misses require a documented tradeoff and backend evidence.
- Same-host loopback percentage reduction is diagnostic only. It is not a
  blocking promotion gate when the 0.3 baseline is already at the loopback
  floor.
- Backend endpoint benchmarks show no regression greater than 5% for
  `http_endpoint_plain_get` and `http_endpoint_path_param_get`.
- `server_fn_json_post` improves by at least 10% in the same-host full backend
  benchmark, or a direct-handler decomposition documents the remaining typed
  route bottleneck.
- Request allocation probes do not regress on the measured integrated
  scenarios.
- The 0.3 to 0.4 migration guide covers all intentionally broken public APIs.

## Stretch Targets

These targets remain useful, but they should not block promotion of the 0.4
performance branch:

- Runtime HTTP median improves by at least 20% on host classes where the 0.3
  baseline remains clearly above the loopback floor.
- `server_fn_json_post` improves by at least 20% after the typed
  server-function route service path replaces the compatibility endpoint path.
- `clean_build_minimal` improves by at least 25% after the minimal feature graph
  split is implemented and measured.
