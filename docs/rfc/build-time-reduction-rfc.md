## 1. Context & motivation

The Reinhardt workspace has grown to **40 crates**, with `reinhardt-core` alone producing a `target/` of 553 MB. Daily development is bottlenecked by build time, especially during the inner edit-compile loop. The pain manifests primarily in three forms (in order of priority):

1. **Incremental build (B)**: editing a file and waiting for `cargo check` / `cargo build` / rust-analyzer.
2. **Cold build (A)**: `cargo clean && cargo build`, fresh CI checkouts, new contributor onboarding.
3. **Test build (C)**: `cargo nextest run --no-run` link phase before tests can execute.

Build performance is also a public-facing concern: any project that depends on `reinhardt = "..."` inherits the framework's compile cost. Improvements to the framework's internal structure propagate to user projects automatically.

A diagnostic pass on the current state surfaced several "free wins" that have not yet been claimed:

- `[profile.dev]` carries only `debug = 1`. No `codegen-units` tuning, no `split-debuginfo`, no `incremental` declaration.
- `.cargo/config.toml` has no linker configuration; macOS uses Apple `ld`, Ubuntu CI uses default `ld`.
- `sccache` is configured globally in `~/.cargo/config.toml`, but **its Rust cache hit rate is 0.32 %** (309 of 309 Rust compile attempts marked `non-cacheable due to: incremental`). It is effectively unused.
- 10 proc-macro crates exist; their generated-code sizes have not been audited.
- `reinhardt-core/target` is 553 MB while its `src/` is 50 KB, which strongly suggests excess monomorphization or large macro output.

This RFC scopes the optimization work into three execution phases (0, A, B) plus a deferred future phase (C). Phases 0/A/B preserve **public API compatibility** to remain RC-safe; Phase C is reserved for breaking changes during the v0.2.0 cycle.

## 2. Goals / non-goals

### 2.1 Primary goal

Reduce inner-loop incremental build time, with cold build and test build improvements as secondary outcomes. Improvements MUST propagate to downstream user projects through framework-internal restructuring (no Cargo.toml changes required from users).

### 2.2 Quantitative targets (against Phase 0 baseline)

| Scenario | Target after Phase A + B |
|---|---|
| `cargo check`, leaf-crate edit (I1) | **−50 % or better** |
| Incremental build, leaf-crate edit (I4) | **−40 % or better** |
| Cold workspace build, default features (B3) | **−25 % or better** |
| CI clean build on `ubuntu-latest` (B1, CI) | **−30 % or better** |
| Cold test build (B4) | **−25 % or better** |

Targets are placeholders; they are recalibrated against the actual Phase 0 baseline.

### 2.3 Non-goals

- **Public API changes.** All work in this RFC preserves the `pub` surface verified by `cargo semver-checks`.
- **Default-feature reduction.** Shrinking `default = ["standard"]` is deferred to Phase C.
- **`watt` adoption.** Pre-compiled WASM proc-macros are deferred to Phase C.
- **Breaking crate splits.** Restructuring that would change crate names or remove crates is deferred to Phase C.
- **Runtime performance changes.** Any runtime change is incidental, not targeted.

### 2.4 Top risks

1. macOS `lld` switch breaks proc-macro / dylib linking. Mitigation: gate behind a spike PR with a measure-then-decide criterion; default remains Apple `ld`.
2. `codegen-units = 256` triggers a latent optimizer-related bug. Mitigation: full `cargo test --workspace --all-features` after the change.
3. sccache strategy revision invalidates existing CI caches. Mitigation: rotate cache key once, document the regeneration cost.

## 3. Considered approaches

| Path | Description | Verdict |
|---|---|---|
| A | Configuration-only quick wins (profile, linker, sccache, GHA cache) | **Selected as Phase A** |
| B | Internal structural adjustments preserving public API (monomorphization, proc-macro thinning, depgraph cleanup, test consolidation) | **Selected as Phase B** |
| C | Default-feature reduction, `watt`, breaking crate splits | **Deferred** to a separate RFC, targeted at v0.2.0 / `develop/0.2.0` |
| Single mega-PR | Bundle A + B into one PR | **Rejected**: prevents per-change effect attribution, blocks rollback granularity |
| Skip Phase A, jump to B | Treat configuration as too small to bother | **Rejected**: A produces measurable improvement at near-zero cost; without A, B's effect on the inner loop is hard to isolate from linker overhead |

## 4. Recommended approach

Execute Phase 0 → Phase A → Phase B as a connected PR series, totaling **15 PRs** (1 + 6 + 8) over ~2–3 weeks. Each PR is independently revertable. Phase C is intentionally out of scope and will be re-brainstormed against the post-Phase-B baseline.

### 4.2 Phase A — Configuration-only quick wins (6 PRs)

All Phase A items are configuration changes. None modify Rust source code. Each PR can be reverted independently.

#### A-1: `[profile.dev]` tuning

```toml
[profile.dev]
debug = "line-tables-only"
split-debuginfo = "unpacked"
codegen-units = 256
incremental = true

[profile.test]
debug = "line-tables-only"
split-debuginfo = "unpacked"
codegen-units = 256
incremental = true
```

Rationale: 256 codegen-units enables maximum parallel codegen of the leaf crate being edited. `line-tables-only` keeps panic backtraces useful while shrinking debuginfo. `split-debuginfo = "unpacked"` is the macOS default, made explicit; on Linux CI it reduces final-link work.

#### A-2: Optimize dependencies in dev profile

```toml
[profile.dev.package."*"]
opt-level = 1

[profile.dev]
opt-level = 0
```

Rationale: dependencies are recompiled rarely; paying a small one-time cost to compile them at `opt-level = 1` makes test/dev runtime substantially faster, shortening the perceived edit-build-test loop. Workspace-internal crates remain at `opt-level = 0` for fast compilation. Acceptance criterion: B1 cold time must not increase by more than 5 %.

#### A-3a: Linker change on Ubuntu CI (`mold`)

```toml
# .cargo/config.toml
[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=mold"]
```

Plus `rui314/setup-mold@v1` immediately after `actions/checkout` in every relevant GHA workflow.

Rationale: `mold` is the strongest available linker for ELF, often producing 5–10× link-time reductions. Expected cold-build delta on CI: 15–30 %.

#### A-3b: Linker spike on macOS Apple Silicon (`ld64.lld`)

A separate spike PR. Adds an opt-in `.cargo/config.lld.toml` selectable via env, runs I1 / I4 with both Apple `ld` and `ld64.lld` five times each, and accepts only if I4 improves by ≥ 20 % or I1 by ≥ 15 %. Otherwise the spike is documented in `docs/build-perf/spikes/lld-macos-2026-05-02.md` and the change reverted.

Rationale: modern Xcode `ld` (since 15.0) is already significantly faster than the older `ld`; the gain from `ld64.lld` may be modest. `mold` does not support Mach-O; `wild`'s Mach-O support is experimental. A measure-first stance is required.

#### A-4: sccache strategy redesign

Diagnosis (recorded for future reference): the current setup wraps `rustc` with sccache while Cargo's `incremental = true` is in effect. The two are incompatible — rustc's incremental state is not cacheable, so sccache marks every Rust invocation `non-cacheable`. Empirical Rust hit rate: 0.32 %.

New configuration:

| Environment | sccache for Rust | `CARGO_INCREMENTAL` | rustc-wrapper |
|---|---|---|---|
| Local development | **off** | `1` (incremental on) | unset |
| CI build / test | **on** | `0` (incremental off) | sccache |

Implementation:

- The repository's `.cargo/config.toml` does NOT set `rustc-wrapper`. Local users keep their personal `~/.cargo/config.toml` settings; documentation advises removing `rustc-wrapper = "sccache"` for Rust there because it does not help when incremental is on.
- All GHA workflows set `CARGO_INCREMENTAL: "0"` and use `mozilla-actions/sccache-action`.
- Cache key derived from `Cargo.lock` hash + Rust toolchain version + OS.
- `sccache --show-stats` is appended to each CI job summary for visibility.

#### A-5: GHA `Swatinem/rust-cache`

```yaml
- uses: Swatinem/rust-cache@v2
  with:
    workspaces: ". -> target"
    cache-targets: true
    cache-on-failure: true
```

This caches `target/` and `~/.cargo/{registry,git}` separately from sccache (which caches per-rustc-output). The two layers compose additively. Total cache size is monitored against the GHA 10 GB-per-repo limit; sub-target directories not on the hot path are excluded via `cache-directories`.

### 4.3 Phase B — Internal structural adjustments (8 PRs)

All Phase B items preserve public API. Each is gated by `cargo semver-checks check-release` plus a full `cargo test --workspace --all-features` pass.

#### B-1: `reinhardt-core` monomorphization reduction

Hypothesis: `target/reinhardt-core` of 553 MB is driven by generic monomorphization and proc-macro output. Analysis:

1. `cargo llvm-lines -p reinhardt-core --release` to enumerate the top-20 generic functions by LLVM-IR line count.
2. Classify each result:
   - **(a) Internal-only generic** → apply the **outline pattern** (small generic shim delegates to a non-generic inner function compiled once).
   - **(b) Public-API-required generic** → leave alone.
   - **(c) `Box<dyn Trait>` would suffice internally** → rewrite the internal implementation while preserving the public signature.

Outline pattern reference example:

```rust
// Before
pub fn register_handler<T: AsRef<str>>(name: T, ...) -> Result<()> {
    // hundreds of lines, monomorphized per T
}

// After (public signature unchanged)
pub fn register_handler<T: AsRef<str>>(name: T, ...) -> Result<()> {
    register_handler_inner(name.as_ref(), ...)
}
fn register_handler_inner(name: &str, ...) -> Result<()> {
    // hundreds of lines, compiled once
}
```

Scope: top-20 results, only those classified (a) or (c). Each function is a separate commit so individual outliers can be reverted.

API-compatibility guard: `cargo semver-checks check-release -p reinhardt-core` plus the full test suite.

#### B-2: Thin proc-macros

Hypothesis: 10 proc-macro crates emit large quantities of repeated code that 30+ dependent crates then compile. Reducing emitted code should reduce both rustc work and the I3 fan-out.

Approach:

1. `cargo expand -p reinhardt-core --tests` (and similar for representative dependents) to inspect actual macro output.
2. Pick macros where output is large and templated. In scope for this RFC: `#[model]`, `#[inject]`, `#[page]` (the highest-volume three).
3. Refactor: macros emit only metadata declarations; runtime helper functions in the runtime crates do the heavy lifting.

Sketch:

```rust
// Before — macro inlines ~200 lines per model
#[model(table = "users")]
struct User { ... }

// After — macro emits metadata only; logic moves into runtime crate
#[model(table = "users")]
struct User { ... }
// Expansion:
//   impl Model for User {
//       const META: ModelMeta = ModelMeta::new(...);
//   }
// Logic lives in `reinhardt-db::model::register_with_meta::<Self>(Self::META)`.
```

Constraints:

- Public macro API (attribute syntax, names of generated `pub` items) unchanged.
- New helper functions in runtime crates are non-breaking additions.
- `watt` is **not** introduced here (Phase C).

API-compatibility guard: `cargo expand` diff appended to PR description; semver-checks.

#### B-3: Critical-path shortening via depgraph cleanup

Hypothesis: among 40 crates, some carry transitive dependencies that exist only to share a single type, blocking parallel build progress.

Approach:

1. Generate `cargo depgraph --workspace --build-deps | dot -Tsvg > depgraph.svg`.
2. Read `cargo build --timings` HTML for the critical path; enumerate the top-3 bottleneck chains.
3. Apply API-preserving fixes:
   - **(p1)** Extract the shared-type subset into a new leaf crate (e.g., `reinhardt-core-types`). The original crate `pub use`s the new crate's items, so user-visible paths are preserved.
   - **(p2)** Convert non-essential transitive deps to `optional = true` behind a feature.
   - **(p3)** Demote candidate dependencies from `[dependencies]` to `[dev-dependencies]` where appropriate.

Example of (p1) preserving the public path:

```rust
// reinhardt-core/src/lib.rs
pub use reinhardt_core_types::ModelMeta;
// User code continues to write `reinhardt::core::ModelMeta` — no source change.
```

Scope: top-3 critical-path chains. Further work flagged as Phase C input.

#### B-4: Integration-test consolidation

Hypothesis: each `tests/*.rs` becomes its own test binary, each requiring its own link. With 40 crates × ~5 test files each, this can dominate `cargo nextest run --no-run`.

Approach:

1. Inventory: `cargo nextest list --workspace`; record per-crate test-binary counts. Collect per-binary build times via `cargo build --tests --timings`.
2. Target crates with ≥ 5 test binaries.
3. Consolidate into a single binary per crate:
   ```rust
   // tests/main.rs
   #![cfg(test)]
   #[path = "auth.rs"]
   mod auth_tests;
   #[path = "model.rs"]
   mod model_tests;
   ```
   ```toml
   [[test]]
   name = "integration"
   path = "tests/main.rs"
   harness = true
   ```

Constraints:

- Test names (the strings used in `cargo nextest run --test ...`) MUST NOT change.
- `serial_test::serial(group)` works correctly inside a single binary; verified during inventory.
- TestContainers state assumptions are inspected per-crate during the inventory PR before consolidation.

Scope: at most 5 crates in this RFC; remaining crates are queued as follow-ups.

## 8. Acceptance criteria

The work described in this RFC is complete when:

1. Phase 0 baseline JSON is committed and the bench script runs reproducibly on macOS Apple Silicon and `ubuntu-latest`.
2. Each Phase A and Phase B PR ships with measurement table in description, against the same baseline.
3. The targets in §2.2 are met. If a target is missed, an explicit trade-off note is added to `docs/build-perf/results-phase-{A,B}.md`.

5. `docs/build-perf/playbook.md` and `instructions/BUILD_PERFORMANCE.md` exist and document the day-to-day workflow plus the prevention guidelines (avoid unbounded monomorphization, keep proc-macro output thin).
