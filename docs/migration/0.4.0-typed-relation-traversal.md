# 0.4.0 Typed Relation Traversal

Reinhardt 0.4.0 adds typed relation paths for ORM relation traversal. Model
derives generate `rel_*` accessors for declared relations, and those paths can
be used in filters, single-valued `select_related`, and direct multi-valued
`prefetch_related`.

## Why migrate

String traversal paths such as `"author__email"` are accepted by compatibility
APIs, but they are not checked until query construction or execution. Typed
paths validate the relation and target field at compile time and also provide a
single join plan for filters and eager loading.

## Filters

Before:

```rust
let posts = Post::objects()
    .filter(F::new("author__email").eq("ada@example.com"))
    .all()
    .await?;
```

After:

```rust
let posts = Post::objects()
    .filter(Post::rel_author().field_email().eq("ada@example.com"))
    .all()
    .await?;
```

## Relation Loading

Before:

```rust
Post::objects().select_related(&["author"]);
Post::objects().prefetch_related(&["comments"]);
```

After:

```rust
Post::objects().select_related(Post::rel_author());
Post::objects().prefetch_related(Post::rel_comments());
```

Typed `select_related` emits the join and selects the related table columns for
single-object relationships. Reverse one-to-many and many-to-many paths are
rejected because a join would duplicate root models; use typed
`prefetch_related` for those paths. Typed `prefetch_related` accepts direct
multi-valued relations, including many-to-many paths that traverse their
generated through table. Forward foreign keys and one-to-one paths must use
`select_related`, because prefetch execution starts from collected root primary
keys rather than foreign-key values. Keep multi-hop prefetch plans split into
explicit queries until full multi-hop prefetch planning is available.

## Optional Traversal

For nullable relations, call `optional()` on the path to request left joins:

```rust
Post::objects()
    .filter(Post::rel_editor().optional().field_email().is_null());
```

## Reverse Relations

Forward relations generate ergonomic `rel_<name>()` methods on the declaring
model. Reverse relations generated only from another model's `related_name` do
not create methods on the target model. If the target model explicitly declares
the reverse relation, the macro also generates the ergonomic reverse method:

```rust
User::objects()
    .filter(User::rel_posts().field_title().icontains("rust"));
```

## Compatibility

String relation APIs remain available in 0.4.0 so applications can migrate
incrementally. Invalid string relation paths now fail at `select_related` /
`prefetch_related` construction when model relationship metadata is available.
Manual `Model` implementations without metadata retain the legacy naming
fallback. Chaining string and typed relation loaders appends and deduplicates
both plans, so migration order does not change the generated queries. Keep simple
same-table expressions such as `F::new("updated_at")` when a typed relation path
is not involved.

Counts over typed reverse or many-to-many filters use distinct root primary
keys so matching multiple related rows does not inflate the result.

Typed related filters are supported for SELECT queries. `update_fields`,
`update_sql`, and `delete_sql` reject them because those write builders do not
emit the joins required by related aliases; use a subquery or a prior SELECT
when a write must depend on related rows.
