# Migrating off `client_validators:` (0.1.0-rc.16 → 0.2.0)

**Applies to:** `form!` macro users on reinhardt-web 0.1.0-rc.16 or newer.
**Tracked by:** [#3654](https://github.com/kent8192/reinhardt-web/issues/3654).

## Why this changed

The `client_validators:` block in the `form!` macro accepted raw JavaScript
expression strings. Investigation in 2026-04 confirmed:

- The JS strings were parsed and stored in the AST, but **never referenced by
  code generation** in any released version. No runtime output was ever
  produced from a `client_validators:` block.
- Zero test or example files in the repository used it.
- JS-string validators were not type-checked by `rustc`, not shared with the
  server `validators:` block, and required context-switching between Rust and
  JavaScript inside a single macro invocation.

Rather than silently removing unused syntax, 0.1.0-rc.16 surfaces the
situation as a customized compile error that points to this guide. 0.2.0
(stable) removes the identifier from the parser entirely; users who have
migrated will not notice the change.

## Before / after

### Before (no longer supported)

```rust
form! {
    name: LoginForm,
    server_fn: login,

    fields: {
        username: CharField { required, min_length: 3 },
        password: CharField { required, widget: PasswordInput, min_length: 8 },
    },

    validators: {
        username: [
            |v| !v.trim().is_empty() => "Username cannot be empty",
        ],
    },

    client_validators: {
        username: [
            "value.length >= 3" => "Username must be at least 3 characters",
        ],
        password: [
            "value.length >= 8" => "Password must be at least 8 characters",
        ],
    },
}
```

### After

```rust
form! {
    name: LoginForm,
    server_fn: login,

    fields: {
        username: CharField { required, min_length: 3 },
        password: CharField { required, widget: PasswordInput, min_length: 8 },
    },

    validators: {
        username: [
            |v| !v.trim().is_empty() => "Username cannot be empty",

            #[client(on = input)]
            |v| v.len() >= 3 => "Username must be at least 3 characters",
        ],
        password: [
            #[client(on = input)]
            |v| v.len() >= 8 => "Password must be at least 8 characters",
        ],
    },
}
```

Each rule is a Rust closure. Rules compile to native code for the server
`.validate()` method. (Client-side event-handler code generation for
`#[client(on = ...)]` rules is delivered in a follow-up; in 0.1.0-rc.16 the
scope annotations are parsed and type-checked but only `#[server]` and the
default `Both` rules are currently emitted by `.validate()`.)

## Scope annotation quick reference

| Annotation | Server (`.validate()`) | Client | Client trigger |
|------------|------------------------|--------|----------------|
| (none) | Yes | Yes | submit |
| `#[server]` | Yes | No | — |
| `#[client(on = submit)]` | No | Yes | submit |
| `#[client(on = input)]` | No | Yes | input |
| `#[client(on = blur)]` | No | Yes | blur |
| `#[server]` + `#[client(on = blur)]` | Yes | Yes | blur |

## FAQ

### Did my client-side validation ever run?

If you only used the macro's `client_validators:` block: **no**. The JS
strings were parsed but never emitted as output. Browsers never received
them.

If you used `form.add_validator_rule("email", ...)` at runtime (against
`ValidationRule::ValidatorRef` and the `ValidatorRegistry` in
`reinhardt-pages::form::validators`): **yes**. That API is unchanged and
remains supported.

### Does this change affect `ValidatorRegistry` / `EmailValidator` / `MinLengthValidator` / `UrlValidator`?

No. Those live in `reinhardt-pages::form::validators` and are reachable via
`form.add_validator_rule(...)`. They are unrelated to the `client_validators:`
macro block being removed.

### What about the WASM target?

Server-side closures compile for the native target. WASM-based client-side
event handlers for `#[client(on = ...)]` rules are a follow-up; the
annotations are already accepted by the parser so you can write
forward-compatible code today.

### When is the hard-error removed?

0.2.0 (next stable) removes the `client_validators` identifier from the
parser. After that, using the block produces a generic `unknown property`
error with no migration hint. Please migrate on 0.1.x.
