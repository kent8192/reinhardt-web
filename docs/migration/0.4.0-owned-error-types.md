# Migrating to Repository-Owned Error Types in 0.4.0

Reinhardt 0.4.0 removes the direct `anyhow` dependency and the opaque
`Error::Other` compatibility variant. Framework and database boundaries now
return named error types that callers can classify without downcasting.

## Construct Database Errors with a Category

Before 0.4.0, database failures carried only a message:

```rust
use reinhardt::core::exception::Error;

let error = Error::Database("email already exists".to_owned());
```

Construct a `DatabaseError` with a portable `DatabaseErrorKind` instead:

```rust
use reinhardt::core::exception::{DatabaseError, DatabaseErrorKind, Error};

let error = Error::from(DatabaseError::new(
    DatabaseErrorKind::UniqueViolation,
    "email already exists",
));
```

Use the most specific available category. `DatabaseErrorKind::Query` is the
fallback for query failures that do not fit a connection, constraint, syntax,
type, transaction, configuration, serialization, or unsupported-operation
category.

## Match Portable Categories

Before 0.4.0, callers had to interpret the database message:

```rust
use reinhardt::core::exception::Error;

fn is_conflict(error: &Error) -> bool {
    matches!(error, Error::Database(message) if message.contains("unique"))
}
```

Match `database_kind()` instead. The enum is non-exhaustive, so include a
fallback arm:

```rust
use reinhardt::core::exception::{DatabaseErrorKind, Error};

fn response_status(error: &Error) -> u16 {
    match error.database_kind() {
        Some(DatabaseErrorKind::UniqueViolation)
        | Some(DatabaseErrorKind::ForeignKeyViolation) => 409,
        Some(DatabaseErrorKind::NotNullViolation)
        | Some(DatabaseErrorKind::CheckViolation) => 400,
        Some(DatabaseErrorKind::Connection)
        | Some(DatabaseErrorKind::Timeout) => 503,
        Some(_) | None => 500,
    }
}
```

## Read Optional Vendor Codes

Database messages were previously the only available diagnostic detail:

```rust
use reinhardt::core::exception::Error;

fn diagnostic(error: &Error) -> Option<&str> {
    match error {
        Error::Database(message) => Some(message.as_str()),
        _ => None,
    }
}
```

Use `database_error()` when vendor-specific handling is unavoidable, and keep
portable behavior based on `database_kind()`:

```rust
use reinhardt::core::exception::Error;

fn vendor_code(error: &Error) -> Option<&str> {
    error.database_error().and_then(|database| database.code())
}
```

Vendor codes are optional and backend-specific. Do not use them as the primary
cross-database control flow.

## Propagate Framework Failures Through Application Errors

Opaque application wrappers previously erased the framework error variant:

```rust
#[derive(Debug)]
struct ApplicationError(Box<dyn std::error::Error + Send + Sync>);
```

Define a named application error and provide
`From<reinhardt::Error>`. Transaction helpers use that conversion for begin,
commit, and rollback failures while preserving domain variants:

```rust
#[derive(Debug, thiserror::Error)]
enum ApplicationError {
    #[error("order was rejected")]
    Rejected,
    #[error(transparent)]
    Framework(#[from] reinhardt::Error),
}
```

This keeps matching explicit and preserves the framework error as the source.
