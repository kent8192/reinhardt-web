# reinhardt-web v0.1.0-rc.13

## Highlights

`InfraSignals` gains seven new detection fields ŌĆö `grpc`, `storage`, `mail`, `session_backend`, `graphql`, `admin_panel`, and `i18n` ŌĆö giving PaaS platforms a much richer picture of an application's infrastructure requirements at analysis time. Each field follows the existing token-based matching pattern, and `session_backend` uses compound token matching specifically to avoid false positives against cache detection.

A significant bug in the dependency injection layer is fixed: the `use_inject` macro previously reused the shared router `InjectionContext` that was created at application startup and never carried the current HTTP request. Injectable types such as `AuthInfo` that call `ctx.get_http_request()` to extract auth headers or cookies would therefore receive `None` and fail with a runtime HTTP 500. The macro now forks a per-request context from the shared one ŌĆö inheriting singleton-scoped dependencies while providing a fresh request scope containing the live `Request` ŌĆö resolving the error without any API changes.

Two migration reliability fixes land in this release. `makemigrations --merge` previously failed with a spurious `DuplicateOperations` error when writing the merge migration file: because merge migrations intentionally carry empty operation lists, the duplicate check's `Vec::new() == Vec::new()` comparison always fired. An empty-operations guard now skips that comparison. Separately, the AST parser's `extract_string_literal` function only matched direct string literals (`"foo"`), silently ignoring the `"foo".to_string()` method-call form that `makemigrations` actually emits for dependency tuples. Unrecognised dependencies meant all migrations had in-degree 0 in topological sort, causing non-deterministic execution order and `relation "X" does not exist` errors at runtime.

## Breaking Changes

No breaking changes in this release.
