# reinhardt-web v0.1.0-rc.24

## Highlights

Two routing bugs that caused silent failures are now fixed. `ServerRouter::mount()` panics early with a clear message when a mount prefix contains path parameter placeholders such as `{org}` — previously the framework accepted the prefix but matched it as a literal string, silently returning 404 for every route under that prefix. Separately, `Path<(T1, T2, ...)>` tuple extraction now populates elements in URL pattern declaration order instead of alphabetically by parameter name, eliminating silent HTTP 400 errors that occurred whenever tuple type order happened not to match alphabetical name order. Both fixes align reinhardt-web's behaviour with Axum and Actix conventions.

A long-standing gap in the migration autodetector is closed: `#[model(unique_together = ...)]` declarations are now reliably detected and emitted as `Operation::AddConstraint` / `Operation::DropConstraint` during `cargo make makemigrations`. The fix spans the full pipeline — the macro now propagates `unique_together` entries into `ModelMetadata`; the autodetector's cross-state model lookups use `table_name` as the key (making them robust against the struct-name vs. table-derived-name skew that occurs during offline state reconstruction); and `generate_migrations()` (the CLI entry point) now mirrors the constraint-emission loops that were previously only present in `generate_operations()`. Projects that previously had to hand-write composite UNIQUE migrations can regenerate them automatically after this release.

Two additional fixes eliminate spurious no-op `AlterColumn` operations emitted for unchanged primary-key fields when `from_state` is rebuilt from migration files. The autodetector now canonicalises schema-affecting field parameters through `ColumnDefinition::from_field_state` before comparison, collapsing the dense-vs-sparse `params` HashMap asymmetry between the registry and migration-replay paths. The `#[model]` macro also stops emitting `null = "true"` for `Option<T>` primary key fields: the `Option` wrapper is a Rust-side convention for pre-insert state and carries no DB-level nullability, so PK fields now correctly advertise `null = "false"` to every downstream consumer of `FieldState`.
