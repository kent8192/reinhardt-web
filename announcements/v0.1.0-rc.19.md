# reinhardt-web v0.1.0-rc.19

## Highlights

This release restores dependency graph integrity across the workspace. `reinhardt-auth-macros` and `reinhardt-streaming` were previously excluded from the `version_group = "reinhardt"` synchronization in `release-plz.toml` and hardcoded their own version strings, leaving them frozen at `0.1.0-rc.16` and `0.1.0-alpha.1` respectively while the rest of the workspace advanced. Because published crates on crates.io still declared `^0.1.0-rc.16` and `^0.1.0-alpha.1` as requirements, macro-side changes in `reinhardt-auth` and streaming improvements used by `reinhardt-tasks`/`reinhardt-urls` were invisible to downstream consumers. Both crates now inherit `version.workspace = true` and participate in workspace-wide version bumps going forward.

Two scaffold bugs affecting `reinhardt-admin startapp --with-pages` are fixed. The generated `ws_urls.rs` template previously produced a `ws_url_patterns()` function that returned `()` instead of a `WebSocketRouter`, causing an `E0599` compile error on every freshly scaffolded pages project. The template now carries the correct return type and a valid `WebSocketRouter` body. Independently, enabling the `admin` feature no longer silently suppresses the `pages` module: `reinhardt::pages::ClientLauncher` and `reinhardt::pages::router::Router` are now reachable for any project activating `admin` alone, without requiring an additional explicit `pages` feature flag.

This release also restructures routing modules for pages-capable apps. `ws_url_resolvers` moves from the app-root `ws_urls` module to `urls::ws_urls`, making server, client, and ws routing modes symmetric under a shared `urls/` directory. The scaffold now emits `urls/server_urls.rs`, `urls/client_urls.rs`, and `urls/ws_urls.rs` as submodules. Existing projects with a top-level `ws_urls.rs` require a one-time migration — see the Breaking Changes discussion for the exact steps.
