# reinhardt-web v0.3.0-rc.2

## Highlights

This release adds a conditional partial-update API to `QuerySet`, allowing callers to issue atomic `UPDATE … SET … WHERE …` statements directly against any filtered row set without materializing or replacing whole model instances. The new `update_fields` execution path exposes typed `FieldAssignment` / `UpdateValue` primitives and `FieldRef::assign(…)` helpers generated alongside model fields. Empty-assignment calls and predicate-less partial updates are rejected at the API boundary, keeping the safety guarantees users already rely on from filtered `QuerySet` operations.

`SessionMiddleware` now automatically derives `AuthState` — including authenticated status and admin/staff flags — from the `USER_ID_SESSION_KEY` already present in `SessionData`. Applications using the standard cookie-session stack no longer need a second auth middleware layer just to satisfy `CurrentUser<U>` extension requirements. Any pre-existing `AuthState` (for example, one populated by a JWT middleware earlier in the chain) is preserved, so custom authentication setups are unaffected.

A bug in the migration autodetector that produced spurious `AlterColumn` and `AddConstraint` operations for replayed auth-model schemas is now fixed. The root cause was a mismatch between how `ProjectState` tracked single-column unique constraints added via `CreateTable` / `AddConstraint` and how field alteration comparisons canonicalized inline-unique versus tracked-UNIQUE representations, UUID primary-key `auto_increment` values, and default-only differences on existing columns. A dedicated regression test covering the auth schema shape reported in #5367 guards against recurrence.

## Breaking Changes

No breaking changes in this release.
