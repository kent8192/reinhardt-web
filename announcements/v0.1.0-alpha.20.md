# reinhardt-web v0.1.0-alpha.20

## Highlights

This release delivers two targeted bug fixes to the command-line startup and database configuration layers, addressing pain points that could cause confusing runtime failures.

**Database management commands no longer require a registered router.** Previously, `execute_from_command_line()` called `auto_register_router()` unconditionally before parsing CLI arguments. This meant that pure database operations—`migrate`, `makemigrations`, and similar commands—would fail with a "No URL patterns registered" error even when no HTTP server was intended. Router registration is now skipped for commands that do not require it, so database workflows function correctly in projects that have no routes defined yet.

**The zero-routes error message now guides users toward the linker workaround.** Projects structured as a standard `src/lib.rs` + `src/bin/manage.rs` split can silently lose `inventory::submit!` side effects due to linker dead-code elimination. The improved error message now includes a concrete hint about the lib/binary linker workaround, turning a cryptic failure into an actionable diagnosis.

**`DATABASE_URL` is now synchronized from settings at `runserver` startup.** The framework has two independent database connection paths: the ORM resolver (which already read `settings/*.toml`) and direct pool consumers (which read the `DATABASE_URL` environment variable). When a database URL was configured only in `settings/local.toml`, the ORM reported "Database: connected" in the startup banner while any code path that read `DATABASE_URL` directly received no value—causing silent authentication failures at request time. The resolved URL is now written to `DATABASE_URL` before any request-handling tasks are spawned, ensuring both paths use the same credentials.

## Breaking Changes

No breaking changes in this release.
