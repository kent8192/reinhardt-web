# reinhardt-web v0.2.0-rc.3

## Highlights

`reinhardt-pages` WASM builds now correctly enable the `security` feature from `reinhardt-core`, resolving compile failures that surfaced when building SPA navigation fixtures and other WASM targets with non-default feature sets. A hydration unknown-event warning emitted in the same code path is also silenced.

A new `routing` feature flag on the `reinhardt-web` facade reduces the dependency footprint for core-only builds. Projects that compile with `--no-default-features` or `--features core` no longer pull in the full URL routing stack. In practice, `facade-core` drops from roughly 661 crates (23 heavy entries) to 368 crates (3 heavy entries). Default, `minimal`, `standard`, `pages`, and `client-router` presets retain routing as before, so existing projects require no changes.

WASM `server_fn` transport is now reliable when using the MSW test harness. Endpoints are resolved against the browser document URL, eliminating `RelativeUrlWithoutBase` errors that previously blocked generated clients. The MSW fetch interceptor is updated to be compatible with reqwest's WASM backend — it now preserves `Response.url`, reads bodies from `fetch(Request)` inputs, and maps mocked `ServerFnError::Server` values to the correct HTTP status and text body format. This makes WASM integration testing for server functions practical without workarounds.

A bug in the tutorial polls example is fixed where selecting a radio input on the vote form could be silently cleared on rerender. The form instance is now held stable for the route component lifetime so `page!` auto-wrapped rerenders no longer recreate the form with an empty `choice_id`.

## Breaking Changes

No breaking changes in this release.
