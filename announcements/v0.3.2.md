# reinhardt-web v0.3.2

## Highlights

This release focuses heavily on security hardening across authentication and data-handling paths. `JwtSessionSettings` now stores its signing secret in a `SecretString`, preventing the HMAC key from leaking through debug output or serialized settings dumps. The default `ErrorBoundary` fallback in `reinhardt-pages` no longer renders tracked resource error text, avoiding accidental disclosure of internal diagnostics to end users. Generic OIDC providers now enforce same-origin discovery, closing an SSRF pivot vector where a malicious or misconfigured discovery document could redirect authentication flows to unrelated hosts. Additionally, `CustomManager`'s `before_save`, `before_delete`, and `before_bulk_update` veto hooks are now actually invoked by the default write paths, ensuring hook-based authorization and validation can no longer be silently bypassed, and request-scoped `AuthIdentity` values correctly propagate the `is_active` flag again so deactivated accounts are properly blocked by `IsActiveUser` permission checks.

On the framework side, `reinhardt-pages` fixes a nested reactive DOM ownership bug where deeply nested `{expr}`/`if`/`for`/`watch` reactive blocks could leave stale content or leaked effects behind when an outer branch changed; nested reactive subtrees are now mounted under a dedicated wrapper so the outer owner can clean them up atomically. The `migrate --plan` command now displays pending migrations in the same dependency-resolved execution order used by real migration runs, so plan previews no longer appear to violate declared cross-app dependencies.

Finally, this release introduces the new `HttpError` trait and `#[derive(HttpError)]` macro, giving applications a declarative way to map error types to HTTP status codes and client-safe messages while automatically preventing 5xx responses from leaking internal error details.

## Breaking Changes

No breaking changes in this release.
