# reinhardt-web v0.1.0-alpha.14

## Highlights

This release delivers three notable improvements to DCL statement generation in `reinhardt-query`, all related to correctness and security.

**Password parameterization in DCL SQL** (PR #268) addresses a high-severity security issue (CVSS 7.5, CWE-532/CWE-312) where passwords were embedded literally in generated SQL strings. DCL statements now emit parameterized placeholders — `?` for MySQL and `$N` for PostgreSQL and CockroachDB — keeping credentials out of query logs, APM traces, query plan caches, and debug output.

**Correct MySQL `user@host` syntax** (PR #267) fixes a formatting bug where MySQL user identifiers were incorrectly rendered with backtick quoting. All affected DCL statements — CREATE USER, ALTER USER, DROP USER, RENAME USER, and SET DEFAULT ROLE — now produce the standard `'user'@'host'` form. When no host is specified, the host defaults to `'%'` (any host), matching MySQL's own convention.

**Standardized empty-string validation across DCL statements** (PR #266) introduces a shared `validate_name()` function in `dcl/mod.rs` that consistently rejects empty and whitespace-only names. Previously, some statements silently accepted invalid input and produced broken SQL. Methods accepting user or role names now return `Result<&mut Self, String>`, and `AlterUserStatement` now requires at least one change to be specified before building.

Note: all three DCL PRs introduce breaking API changes. Callers must update to handle `Result`-returning methods and ensure names are non-empty after trimming. Refer to the individual PR descriptions for migration examples.

## Breaking Changes

No breaking changes in this release.
