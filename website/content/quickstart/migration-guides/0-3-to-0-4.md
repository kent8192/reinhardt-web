# Migrating from 0.3.x to 0.4.0

Reinhardt 0.4 adds native model enum fields. A model enum owns an explicit
database codec, participates in typed filters and updates, and contributes a
named check constraint to generated migrations.

This guide focuses on replacing `String` fields with enums and on changing an
existing enum's values or representation safely.

## Define the Persistent Contract

Choose either string or `i32` storage and assign every variant an explicit
database value:

```rust
use reinhardt::ModelEnum;
use reinhardt::core::serde::{Deserialize, Serialize};

#[derive(ModelEnum, Clone, Debug, PartialEq, Serialize, Deserialize)]
#[model_enum(repr = "string")]
enum Status {
	#[model_enum(value = "queued")]
	Queued,
	#[model_enum(value = "in_progress")]
	Running,
}

#[derive(ModelEnum, Clone, Debug, PartialEq, Serialize, Deserialize)]
#[model_enum(repr = "i32")]
enum Priority {
	#[model_enum(value = 10)]
	Low,
	#[model_enum(value = 20)]
	High,
}
```

These declarations create three independent contracts:

- `Running` is the Rust variant name.
- A serde name such as `#[serde(rename = "RUNNING")]` is the serialized API value.
- `"in_progress"` is the persistent database value.

Changing one does not change the others. Treat `#[model_enum(value = ...)]` as
schema and data, not as display text.

## Replace a String Field Without Rewriting Values

Suppose a 0.3 model stores `"queued"` and `"in_progress"` in a `String` field:

```rust,ignore
#[field(max_length = 32)]
status: String,
```

Replace it with the string-backed enum while preserving `max_length`,
`db_column`, and nullability attributes:

```rust,ignore
#[field(max_length = 32)]
status: Status,
```

If every existing row already contains one of the explicit enum values, the
physical representation and stored bytes are unchanged. The migration does not
need a data rewrite; it adds the generated enum-domain check constraint. Review
the generated migration before applying it, because any undeclared value that
already exists will cause the new constraint to fail.

For a nullable field, use `Option<Status>` and keep `null = true`:

```rust,ignore
#[field(max_length = 32, null = true)]
fallback_status: Option<Status>,
```

SQL `NULL` remains valid. Non-null values must be declared by the enum.

## Update Queries and Partial Updates

Field references now require the model enum type. Replace raw strings and
integers with variants:

```rust,ignore
let jobs = Job::objects()
	.filter(Job::field_status().eq(Status::Queued))
	.filter(Job::field_priority().is_in([Priority::Low, Priority::High]))
	.all()
	.await?;

Job::objects()
	.filter(Job::field_id().eq(job_id))
	.update_fields([
		Job::field_status().assign(Status::Running),
		Job::field_fallback_status().assign(Some(Status::Queued)),
	])
	.await?;
```

Code such as `Job::field_status().eq("queued")` now fails to compile because
`&str` does not implement the field-value contract for `Status`. This is
intentional: create, hydrate, filter, membership, and assignment operations all
use the same database codec.

## Change Values or Representation Safely

Changing a persistent value, removing a variant, or switching between
`repr = "string"` and `repr = "i32"` requires a data migration. The migration
autodetector reports these transitions and generates the schema operations, but
it cannot infer how application data should map.

Use this order:

1. Inventory distinct stored values, including values written outside the ORM.
2. Define an explicit mapping from every old value to a new value.
3. Drop or defer the old enum-domain check before writing values it rejects.
4. Run the data conversion while the column still has a representation that can
   express the conversion, using backend-appropriate migration SQL or code.
5. Alter the physical column type when the representation changes.
6. Add the new generated enum-domain check only after every row is valid.
7. Deploy application code that reads and writes the new contract in the same
   coordinated release.

Do not simply edit `#[model_enum(value = ...)]` and apply the generated check.
For a representation change, review conversion syntax for every enabled
database backend; string-to-integer casts and constraint-alteration behavior are
backend-specific.

Reordering variants without changing their explicit values is a schema no-op.
The persistent values, not declaration order, define the domain.

## Generated Constraints and Hydration Errors

Generated migrations add a named check constraint for every required or
nullable enum field on PostgreSQL, MySQL, and SQLite. String values are quoted
as strings, `i32` values remain numeric, and nullable fields continue to accept
`NULL`.

The check protects writes made through SQL as well as ORM writes. It is still
possible to encounter legacy invalid data when constraints were disabled,
added later, or bypassed. Hydrating an undeclared value returns a
`FieldCodecError::InvalidEnumValue` containing the model, Rust field, resolved
database column, representation, and stored value. Reinhardt does not choose a
fallback variant automatically.

## Verification

After updating the model and migration:

```bash
cargo make makemigrations
cargo make migrate
cargo check --all-features
cargo test --doc
```

Inspect the migration operations and generated SQL in addition to running the
commands. Test at least one create/hydrate round trip, `eq`, `is_in`, `assign`,
`None` and `Some` for nullable fields, and rejection of a value outside the
declared domain.
